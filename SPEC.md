# 医疗内容OS（Content OS）开发规格文档

## 📋项目概述

**项目名称**：医疗新媒体内容生产操作系统（Content OS）

**开发模式**：分阶段稳定迭代开发（STEP 1 → STEP 10）

**核心目标**：构建基于 Gemini 2.5 Flash + RAG 的医疗内容生成系统，用于糖尿病/减重/营养领域内容生产。

---

## ⚠️开发规则

| 规则 | 说明 |
|------|------|
| ❌禁止跳步 | 必须按 STEP 1~10 顺序执行 |
| ❌禁止提前实现 | 每步仅实现当阶段功能，不做超前优化 |
| ✅可运行验证 | 每步完成后必须输出测试结果 |
| ✅文档记录 | 每步需记录实现内容与验证结果 |

---

## 🧩分阶段功能规格

### STEP 1：基础系统骨架

**目标**：建立最小可运行系统

**功能**：
- 提供文本输入接口
- 返回固定模板输出（不调用AI）

**输出示例**：
```
标题：测试内容
正文：系统运行正常
```

**验证标准**：输入任意文本，返回格式化的固定输出

---

### STEP 2：接入 Gemini 内容生成

**目标**：实现AI写作能力

**功能**：
- 接入 Gemini 2.5 Flash API
- 输入选题 → 输出文章草稿
- 生成完整中文科普文章

**验证标准**：输入选题，输出完整中文文章

---

### STEP 3：智能内容生成（按需适配）

**目标**：实现灵活的内容生成能力

**功能**：根据用户指令动态生成内容：

| 指令示例 | 输出要求 |
|----------|----------|
| "根据xxxx，生成一篇糖尿病科普文章1000-1500字" | 完整科普文章 |
| "根据最新消息xxxxx，生成一条300-600字的减肥科普抖音脚本" | 抖音口播脚本 |
| "根据xxxx，生成一篇小红书笔记" | 小红书格式内容 |

**核心能力**：
- 自动识别内容类型（文章/脚本/笔记）
- 自动识别字数要求
- 自动识别领域（糖尿病/减重/营养）

**验证标准**：根据不同指令格式，生成对应格式和长度的内容

---

### STEP 4：文献引用系统（RAG基础）

**目标**：让内容具备医学依据，支持查看原文

**功能**：
- 建立文献库（PMID / DOI / 摘要 / 原文链接）
- 根据关键词检索相关文献
- 在文章中插入引用
- **引用可点击，点击后打开原文献页面**

**输出格式**：
```
结论：xxx
参考文献：PMID:xxxxx / DOI:xxxxx [查看原文](链接)
```

**验证标准**：
- 生成内容包含相关文献引用
- 引用链接可点击访问原文献

---

### STEP 5：医学知识库接入

**目标**：增强医学准确性

**功能**：
- 导入科普文章库
- 根据选题检索相关知识
- 辅助生成内容

**验证标准**：内容体现知识库中的专业知识

---

### STEP 6：用户认知库（评论数据）

**目标**：提升内容"人味"

**功能**：
- 导入用户评论数据
- 提取高频问题
- 在生成内容中优先体现

**验证标准**：内容反映用户真实关注点

---

### STEP 7：平台表达适配（爆款仿写）

**目标**：通过学习爆款内容实现风格适配

**功能**：
- 用户提供各平台爆款内容示例
- 系统自动解析爆款文的风格、逻辑结构、语言特点
- 根据解析结果仿写新内容

**解析维度**：
| 维度 | 说明 |
|------|------|
| 标题风格 | 悬念型/冲突型/数字型/疑问型 |
| 开篇逻辑 | 冲突开场/痛点切入/故事引入 |
| 正文结构 | 段落数/每段逻辑/论证方式 |
| 语言特点 | 口语化程度/情绪词/专业术语比例 |
| 结尾方式 | 呼吁行动/情感共鸣/总结升华 |

**输入示例**：
```
# 爆款内容库
[用户提供抖音爆款脚本、小红书爆款笔记、公众号爆款文章]

# 仿写指令
根据这些爆款风格，生成一条关于GLP-1的抖音脚本
```

**验证标准**：
- 成功解析爆款内容特征
- 仿写内容体现学习到的风格特点

---

### STEP 8：爆款结构（轻量版）

**目标**：提升传播性

**固定结构**：
```
冲突 → 误区 → 解释 → 证据 → 建议
```

**验证标准**：所有内容自动套用此结构

---

### STEP 9：行业新闻科普生成

**目标**：支持热点内容生成

**输入**：新闻/政策/医学研究

**输出结构**：
- 标题（新闻型）
- 核心结论
- 分点解读
- 临床意义
- 注意事项
- 文献来源

**验证标准**：新闻类输入生成对应结构化输出

## 🎯最终系统行为

### 自然语言指令输入
```
# 文章生成
根据GLP-1最新研究进展，生成一篇1500字的糖尿病科普文章

# 脚本生成
根据最新消息xxxxx，生成一条300-600字的减肥科普抖音脚本

# 笔记生成
根据糖尿病饮食指南，生成一篇小红书笔记
```

### 输出内容
| 输出项 | 说明 |
|--------|------|
| 生成内容 | 根据指令类型生成对应格式 |
| 文献引用 | 带可点击链接，查看原文献 |
| 字数统计 | 显示实际生成字数 |

## 🛠️技术架构

### 基础架构（原始）

```
┌─────────────────────────────────────────┐
│           自然语言指令层                 │
│  (文章/脚本/笔记 + 字数 + 领域)         │
└─────────────────┬───────────────────────┘
                  │
┌─────────────────▼───────────────────────┐
│           RAG 检索层                     │
│  (文献库 / 知识库 / 认知库)              │
└─────────────────┬───────────────────────┘
                  │
┌─────────────────▼───────────────────────┐
│           AI 生成层                     │
│  (Gemini 2.5 Flash)                     │
└─────────────────┬───────────────────────┘
                  │
┌─────────────────▼───────────────────────┐
│           处理层                         │
│  (平台适配 / 结构优化)                  │
└─────────────────┬───────────────────────┘
                  │
┌─────────────────▼───────────────────────┐
│           输出层                         │
│  (内容 + 文献引用链接)                   │
└─────────────────────────────────────────┘
```

### 增强架构：智能分层知识路由系统（2026-05-12）

```
用户输入
    ↓
┌─────────────────────────────────────────┐
│  Step 1: 意图分析层 (intentAnalyzer.js) │
│  - 识别意图类型：科普/新闻/机制/对比/指南 │
│  - 提取关键词和主题                      │
│  - 判断医学相关性                        │
└─────────────────┬───────────────────────┘
                  ↓
┌─────────────────────────────────────────┐
│  Step 2: 知识路由器 (knowledgeRouter.js)│
│  - 根据意图分配检索权重                  │
│  - 并行调用多源检索                      │
└─────────────────┬───────────────────────┘
                  ↓
    ┌─────────────┼─────────────┐
    ↓             ↓             ↓
┌────────┐  ┌────────┐  ┌────────────┐
│本地知识库│  │ PubMed │  │ 学术搜索   │
│(Qdrant) │  │  API   │  │ (SerpAPI) │
└────┬────┘  └────┬───┘  └─────┬──────┘
    └─────────────┼─────────────┘
                  ↓
┌─────────────────────────────────────────┐
│  Step 3: 信息融合层 (infoFusion.js)     │
│  - 语义去重（相似度 > 0.85 合并）        │
│  - 权重计分（来源 × 时效性 × 权威性）   │
│  - 重新排序                             │
└─────────────────┬───────────────────────┘
                  ↓
┌─────────────────────────────────────────┐
│  Step 4: LLM 生成层                      │
│  - 构建上下文 Context                    │
│  - 意图适配的 Prompt                    │
│  - Gemini/GPT 生成                      │
└─────────────────┬───────────────────────┘
                  ↓
┌─────────────────────────────────────────┐
│  Step 5: 合规检查层 (complianceChecker) │
│  - 敏感词过滤（绝对化表述）              │
│  - 医疗准确性核查（剂量/疗程）          │
│  - 引用来源检查                         │
│  - 风险等级评估                         │
└─────────────────┬───────────────────────┘
                  ↓
┌─────────────────────────────────────────┐
│  多平台输出                              │
│  (微信公众号/抖音/小红书/知乎)          │
└─────────────────────────────────────────┘
```

### 模块说明

| 模块 | 文件 | 功能 |
|------|------|------|
| 意图分析 | `intentAnalyzer.js` | 识别用户意图类型、提取关键词、判断医学相关性 |
| 知识路由 | `knowledgeRouter.js` | 根据意图分配多源检索权重、协调并行检索 |
| 多源检索 | `multiSourceRetriever.js` | 封装 PubMed/学术搜索/新闻检索实现 |
| 信息融合 | `infoFusion.js` | 去重、评分、排序、上下文构建 |
| 合规检查 | `complianceChecker.js` | 敏感词、医疗数据、引用来源检查 |
| 智能工作流 | `smartContentPipeline.js` | 串联所有模块的统一入口 |

### 意图类型与检索权重

| 意图类型 | 本地知识库 | PubMed | 学术搜索 | 新闻 |
|----------|-----------|--------|---------|------|
| 科普 (science) | 0.4 | 0.3 | 0.2 | 0.1 |
| 新闻 (news) | 0.2 | 0.3 | 0.2 | 0.3 |
| 机制 (mechanism) | 0.3 | 0.5 | 0.2 | 0.0 |
| 对比 (comparison) | 0.4 | 0.3 | 0.3 | 0.0 |
| 指南 (guide) | 0.6 | 0.1 | 0.1 | 0.2 |
