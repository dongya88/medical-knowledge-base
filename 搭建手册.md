# 医疗内容OS - RAG知识库系统搭建手册

## 一、项目概述

### 1.1 项目背景

医疗内容OS是一个基于RAG（检索增强生成）技术的智能医疗内容生成系统，主要用于：
- 生成专业的医疗科普文章
- 自动检索和引用权威医学文献
- 确保参考文献的准确性和可追溯性

### 1.2 技术架构

| 组件 | 技术选型 | 说明 |
|------|---------|------|
| 后端服务 | Node.js + Express | 主服务器，处理API请求 |
| 向量数据库 | Qdrant Cloud | 云端存储文档向量 |
| Embedding服务 | 本地 BGE-small | Flask服务，端口5000，向量维度512 |
| 文献检索 | PubMed API | 作为补充知识源 |
| 前端界面 | HTML/CSS/JS | 用户交互界面 |
| 内容生成 | 智谱AI (GLM-4-flash) | 生成文章内容 |

### 1.3 目录结构

```
Content OS/
├── server.js                      # 主服务器
├── embedding_service_small.py      # 本地Embedding服务 (BGE-small)
├── public/                        # 前端文件

│   ├── index.html                 # 页面模板
│   └── style.css                  # 样式文件

│   ├── knowledgeBaseRAG.js        # 知识库RAG类
│   ├── qdrantService.js           # Qdrant向量数据库服务
│   ├── embeddingService.js        # Embedding调用服务
│   ├── multiSourceRetriever.js    # PubMed检索服务
│   └── literatureService.js       # 文献处理服务
├── zhishiku-git/literature/       # PDF文献库目录
├── .env                           # 环境变量配置
├── package.json                    # 依赖管理
└── 需求规格说明书.md               # 需求规格文档
```

### 2.3 重要警告：C盘空间问题

**问题**：HuggingFace模型默认缓存在 `C:\Users\.cache`，可能爆掉C盘导致系统崩溃。

**解决**：设置环境变量将缓存改到D盘
```powershell
setx HF_HOME "D:\hf_models"
setx TRANSFORMERS_CACHE "D:\hf_models"
mkdir D:\hf_models
```

## 六、遇到的问题及解决方案

### 6.1 服务启动问题

**问题**：端口3000被占用，服务器无法启动

**解决**：编写批处理命令自动终止占用进程
```bash
taskkill /F /IM node.exe 2>$null; Start-Sleep -Seconds 2; node server.js
```

### 6.2 环境变量加载失败

**问题**：dotenv模块不可用，导致API Key无法读取

**解决**：手动解析.env文件
```javascript
const envContent = readFileSync(envPath, 'utf-8');
envContent.split('\n').forEach(line => {
    const [key, ...vals] = line.split('=');
    if (key && vals.length) process.env[key.trim()] = vals.join('=').trim();
});
```

### 6.3 参考文献格式错误

**问题**：AI生成的参考文献与正文引用不对应

**解决**：
1. 实现parseAndRenumberReferences函数，按正文出现顺序重排编号
2. 使用Map数据结构实现去重
3. 前端根据result.references数组渲染参考文献列表

### 6.4 知识库搜索失败

**问题**：Qdrant返回"Bad Request"错误

**原因**：Embedding API调用失败（API Key无效或服务不可用）

**解决**：切换到Hugging Face免费Inference API

### 6.5 本地BGE-M3模型跑不动

**问题**：i7-7500U + 8GB内存 加载BGE-M3极慢甚至崩溃

**分析**：
- BGE-M3 参数量567M，需要约8GB内存
- i7-7500U是低压CPU，不是高性能
- C盘只剩22GB，模型缓存可能爆盘

**解决**：改用轻量级的 bge-small-zh-v1.5

### 6.6 文章分段显示问题

**问题**：生成的文章没有分段，所有内容挤在一起

**解决**：前端按换行符分割
```javascript
const paragraphs = mainContent.split(/\n\n+/);
```

### 6.7 引用角标无法跳转

**问题**：点击正文中的[1][2]无法跳转到参考文献

**解决**：实现点击事件处理
```javascript
document.querySelectorAll('.citation-ref').forEach(el => {
    el.addEventListener('click', () => {
        const refNum = el.dataset.ref;
        const refEl = document.getElementById(`ref-${refNum}`);
        refEl.scrollIntoView({ behavior: 'smooth', block: 'center' });
    });
});
```

### 6.8 PubMed文献解析错误

**问题**：XML解析时不同文章的元数据混淆

**解决**：先按`<PubmedArticle>`分割，再逐个提取元数据

### 6.9 知识库被删除问题

**问题**：每次重启服务器，知识库数据都被删除

**原因**：`knowledgeBaseRAG.js`中`forceRecreate`默认值为`true`

**解决**：将`forceRecreate`默认值改为`false`
```javascript
// knowledgeBaseRAG.js 第20行
async initialize(forceRecreate = false) {
    // ...
}
```

### 6.10 引用重复问题

**问题**：生成的文章中参考文献列表有重复条目

**原因**：LLM自行生成了引用列表，与后端处理逻辑冲突

**解决**：
1. 修改提示词，禁止LLM自行生成文献列表
2. 后端统一处理引用编号和文献列表
3. 前端只使用API返回的`references`数组渲染

### 6.11 角标与文献不匹配

**问题**：正文角标数量与参考文献数量不一致

**原因**：`parseAndRenumberReferences`函数逻辑有bug

**解决**：重写函数逻辑：
- 只处理正文中的角标，不处理参考文献列表
- 基于文献标题去重
- 按首次出现顺序重新编号

### 6.12 AI编造内容问题

**问题**：AI生成的内容包含文献中没有的数字或建议

**原因**：提示词约束不够严格

**解决**：优化提示词设计（见第八章提示词设计规范）

### 6.13 API余额不足

**问题**：Jina API报错"Insufficient account balance"

**解决**：切换到本地BGE-small方案，完全免费

## 十、重要注意事项

### 10.1 知识库保护（最重要）⚠️

**核心原则：用户上传的文献永不丢失**

| 操作 | 风险 | 保护措施 |
|------|------|---------|
| 服务器重启 | 无风险 | `forceRecreate=false`，不删除 |
| 修改代码 | 低风险 | 不影响Qdrant数据 |
| 删除集合 | 高风险 | 必须显式传`forceRecreate=true` |

**关键配置**：
```javascript
// knowledgeBaseRAG.js 第20行
async initialize(forceRecreate = false) {  // 必须是false
    // ...
}
```

### 10.2 向量维度兼容性 ⚠️ 非常重要

**核心原则：Embedding 一旦写入 Qdrant，就"绑定死了"**

| 特性 | 说明 |
|------|------|
| 向量维度固定 | 1024 或 512，一旦写入无法更改 |
| 不能混用 | 不同 Embedding 模型产生的向量不能混用 |
| 换模型 = 必须重建 | 换 Embedding 模型 = 清空 Qdrant + 重新上传所有 PDF |

**⚠️ 重点：换模型必须做两件事**
1. 清空 Qdrant 现有索引（删除 collection）
2. 重新上传所有 PDF（重新生成向量）

**当前状态**：
- 已上传约50个 PDF（使用 BGE-small 512维度）
- 知识库记录约5543条
- 不建议切换 Embedding 方案

## 十一、提示词设计规范

### 11.1 设计原则

提示词是控制AI生成内容质量的核心，必须遵循以下原则：

| 原则 | 说明 |
|------|------|
| 明确边界 | 告诉AI只能使用哪些信息 |
| 禁止事项 | 明确列出禁止AI做的事情 |
| 格式要求 | 明确输出格式和结构 |
| 一致性 | 角标编号必须与文献列表一致 |

### 11.2 当前使用的提示词

```javascript
const prompt = `你是医疗科普文章专家，请生成一篇公众号科普文章，主题：${topic}

知识库内容（只包含以下检索到的内容片段）：
${kbContext || '（无）'}

可用文献（共${allReferences.length}条）：
${refInfo}

【核心原则 - 严格基于文献】
1. 【只写文献中有的】只能使用上述文献中明确包含的内容，绝不编造
2. 【禁止无据引用】如果某个具体说法（如食物名称、数字、建议等）不在文献中，必须删除
3. 【不确定就删除】对文献中不明确的内容，坚决不写，只保留有据可查的内容
4. 【数字必须有据】所有数字、剂量、时间等必须来自文献

【格式要求 - 必须使用角标】
5. 正文每个引用处必须使用角标格式：[1]、[2]、[3] 等
6. 在陈述指南建议、研究发现、数据等时，必须在旁边标注对应文献编号
7. 【禁止】不要在文末列出"引用格式"、"参考文献"、"引用列表"等
8. 文献列表由系统自动根据角标生成，你只需在正文中使用角标
9. 同一文献多次引用编号一致
10. 优先引用知识库文献（编号1-${kbReferences.length}），其次引用PubMed

文章结构：开头引入 → 分点正文 → 总结

请生成文章（必须在正文中使用角标[1][2]等标注来源）：`;
```

### 11.3 提示词演进历史

| 版本 | 变化 | 问题 |
|------|------|------|
| 第一版 | 要求使用角标，允许LLM生成文献列表 | LLM生成的文献列表与实际引用不一致 |
| 第二版 | 强调禁止编造数字 | 太严格，AI不敢使用任何引用 |
| 第三版 | 禁止LLM生成文献列表，强调后端统一处理 | 当前使用 |

### 11.4 内容质量检查清单

生成文章后，检查以下问题：

- [ ] 每个数字旁边是否有角标？
- [ ] 引用的文献是否确实包含该数字/内容？
- [ ] 是否有发现AI编造的内容？
- [ ] 同一文献多次引用编号是否一致？
- [ ] 角标数量是否等于文献列表数量？
- [ ] 文献列表是否有重复条目？

## 十二、故障排查

| 错误信息 | 原因 | 解决方案 |
|----------|------|----------|
| port 3000 is already in use | 端口被占用 | `taskkill /F /IM node.exe` 然后重启 |
| ZHIPU_API_KEY not configured | 环境变量未加载 | 检查.env文件 |
| Bad Request | Embedding失败 | 检查Embedding服务是否运行 |
| Insufficient balance | Jina余额不足 | 已切换到本地BGE-small |
| Knowledge base search failed | Qdrant连接失败 | 检查网络和认证 |
| 模型加载极慢/崩溃 | 内存不足 | 使用bge-small而非bge-m3 |
| TextEncodeInput must be Union | PDF含特殊字符 | 跳过该文件 |
| Invalid PDF structure | PDF结构损坏 | 跳过该文件 |
