# 医疗内容OS - 需求规格说明书 (SRS)

> 本文档是医疗内容OS RAG知识库系统的核心需求规格，定义了系统的设计目标、功能要求、质量标准和呈现规范。

---

## 一、系统概述

### 1.1 项目定位

**系统名称**：医疗内容OS - RAG知识库系统

**核心功能**：基于RAG（检索增强生成）技术的智能医疗内容生成系统

**目标用户**：医疗内容创作者、健康管理从业者

### 1.2 核心价值

1. **内容准确性**：所有生成内容必须基于用户上传的真实医学文献
2. **引用可溯源**：每个论断都有明确的文献来源支持
3. **数据完整性**：用户上传的文献不会被系统删除或覆盖

## 三、功能规格

### 3.1 核心功能列表

| 功能ID | 功能名称 | 描述 |
|--------|---------|------|
| F-001 | PDF文献上传 | 将用户PDF文献解析并向量化存储到知识库 |
| F-002 | 智能内容生成 | 根据用户输入主题，检索相关文献并生成科普文章 |
| F-003 | 引用角标渲染 | 在正文中插入可点击的文献角标 |
| F-004 | 参考文献生成 | 自动从正文章节提取角标并生成文献列表 |
| F-005 | 知识库检索 | 支持对已上传文献进行语义搜索 |

### 3.2 功能详细说明

#### F-001: PDF文献上传

**流程**：
1. 用户指定PDF文件路径
2. 系统解析PDF文本内容
3. 将文本分块（chunks）
4. 调用Embedding服务生成向量
5. 存储向量到Qdrant数据库

**数据保护**：
- ⚠️ **关键约束**：`forceRecreate` 参数默认值为 `false`
- 禁止在服务器重启时自动删除知识库
- 禁止覆盖已存在的集合

**失败处理**：
- 文件解析失败：跳过该文件，继续处理下一个
- Embedding服务不可用：返回错误，不删除已有数据
- 常见失败原因：
  - PDF结构损坏（"Invalid PDF structure"）
  - 文本编码问题（"TextEncodeInput must be Union..."）

#### F-002: 智能内容生成

**流程**：
1. 用户输入主题（如"糖尿病患者如何运动"）
2. 系统从知识库检索Top 20相关文献块
3. 收集文献的完整metadata（标题、来源等）
4. 构建提示词，连同文献内容发送给智谱AI
5. AI生成文章内容
6. 后端处理引用编号（去重、重新排序）
7. 返回完整文章及文献列表

**提示词设计原则**：
- 明确告知AI只使用检索到的文献内容
- 禁止AI编造文献中不存在的内容
- 明确要求在正文中使用角标格式 `[1]`、`[2]` 等
- 禁止AI自行生成文献列表（由后端统一生成）

#### F-003: 引用角标渲染

**前端渲染规则**：
1. 将正文中的 `[数字]` 匹配为可点击的 `<span>` 元素
2. 角标样式：蓝色、下划线、鼠标指针
3. 点击角标时：
   - 滚动到页面下方对应文献处
   - 高亮显示2秒
4. 角标 `data-ref` 属性指向文献序号

#### F-004: 参考文献生成

**后端处理流程**：
1. 提取正文中的所有 `[数字]` 角标
2. 将角标映射回原始文献
3. 去重（同一文献只保留一条）
4. 按首次出现顺序重新编号
5. 生成格式化文献列表

**去重规则**：
- 基于文献标题去重
- 保留首次出现的编号

## 四、呈现规格（核心设计规范）

### 4.1 文章结构规范

```
标题：xxx
开头引入：xxx（1-2段背景介绍）

分点正文：
1. xxx [角标]
2. xxx [角标]
   - xxx [角标]
   - xxx [角标]
3. xxx [角标]

总结：xxx（1段）

参考文献：
[1] 文献标题1
[2] 文献标题2
[3] 文献标题3
```

### 4.2 引用角标规范

| 要求 | 说明 |
|------|------|
| 格式 | `[数字]`，如 `[1]`、`[2]`、`[3]` |
| 位置 | 紧跟在引用的内容后面 |
| 可点击 | 是（前端实现） |
| 颜色 | 蓝色 `#007bff` |
| 样式 | 下划线 |

### 4.3 参考文献列表规范

| 要求 | 说明 |
|------|------|
| 编号 | 从 `[1]` 开始连续编号 |
| 格式 | `[编号] 标题` |
| 来源标注 | 显示"知识库"或"PubMed" |
| PMID | 如有则显示并提供链接 |
| 去重 | 同一文献不重复出现 |

### 4.4 一致性要求

| 检查项 | 一致性要求 |
|--------|-----------|
| 角标数量 | 正文中出现的角标编号 ≤ 文献列表数量 |
| 编号连续 | 文献编号从1开始连续，不间断 |
| 无重复文献 | 文献列表中同一文献只出现一次 |
| 角标可点击 | 点击角标能跳转到对应文献 |
| 文献可点击 | 文献列表中的链接可打开原文 |

## 五、质量标准

### 5.1 内容质量标准

| 标准 | 要求 |
|------|------|
| 禁止编造数字 | 所有数字（剂量、时间、次数等）必须来自文献 |
| 禁止无据引用 | 只能引用检索结果中明确存在的内容 |
| 禁止推测 | 不能根据文献内容进行推测或延伸 |
| 不确定则删除 | 对文献中不明确的内容坚决不写 |

### 5.2 数据保护标准

| 标准 | 要求 |
|------|------|
| 知识库持久化 | 服务器重启不删除已有数据 |
| 向量维度匹配 | Embedding维度 = Qdrant向量维度 = 512 |
| 手动删除确认 | 删除集合必须显式传入 `forceRecreate=true` |

### 5.3 系统稳定性标准

| 标准 | 要求 |
|------|------|
| Embedding服务 | 长时间运行不中断 |
| 上传中断处理 | 支持断点续传，不重复上传 |
| 错误隔离 | 单个文件失败不影响整体流程 |

### 6.1 已解决问题

## B.4 账号管理

### B.4.1 初始化账号

系统首次启动时自动创建以下默认账号：

| 用户名 | 密码 | 角色 |

## B.6 API接口设计

### B.6.1 登录接口

**端点**：`POST /login`

**请求参数**（JSON格式）：

```json
{
  "username": "admin",
  "password": "123456"
}
```

**成功响应**（HTTP 200）：

```json
{
  "success": true,
  "user": {
    "id": 1,
    "username": "admin"
  }
}
```

**失败响应**（HTTP 401）：

```json
{
  "success": false,
  "error": "用户名或密码错误"
}
```

**业务逻辑**：

1. 接收username和password参数
2. 从数据库查询username对应的用户记录
3. 使用bcrypt.compare()校验密码
4. 密码正确则写入session：`req.session.user = { id, username }`
5. 返回成功响应

### B.6.2 退出登录接口

**端点**：`POST /logout`

**请求参数**：无

**成功响应**（HTTP 200）：

```json
{
  "success": true,
  "message": "已退出登录"
}
```

**业务逻辑**：

1. 调用 `req.session.destroy()`
2. 返回成功响应

### B.6.3 获取当前用户信息

**端点**：`GET /me`

**成功响应**（已登录，HTTP 200）：

```json
{
  "authenticated": true,
  "user": {
    "id": 1,
    "username": "admin"
  }
}
```

**未登录响应**（HTTP 401）：

```json
{
  "authenticated": false,
  "error": "未登录"
}
```

**业务逻辑**：

1. 检查 `req.session.user` 是否存在
2. 存在则返回用户信息
3. 不存在则返回未登录状态

### B.6.4 核心生成接口（需登录保护）

**端点**：`POST /api/generate`

**登录验证**：必须，未登录返回401

**未登录响应**（HTTP 401）：

```json
{
  "error": "未登录，禁止访问"
}
```

**已登录处理**：正常执行业务逻辑

## B.7 安全控制

### B.7.1 API限流

**依赖包**：`express-rate-limit`

**配置参数**：

```javascript
const apiLimiter = rateLimit({
  windowMs: 60 * 1000,    // 1分钟窗口
  max: 20,                // 最多20次请求
  message: {
    error: "请求过于频繁，请稍后再试"
  },
  standardHeaders: true,  // 返回标准的RateLimit-*头
  legacyHeaders: false    // 禁用X-RateLimit-*头
});
```

**应用范围**：`/api/*` 路由

**说明**：
- 每个IP地址每分钟最多20次API调用
- 防止误刷或恶意调用
- 5-10人正常使用完全足够

### B.7.2 API Key保护

**强制要求**：

| 规则 | 说明 |
|------|------|
| 后端保存 | 所有AI API Key必须存储在server.js或.env中 |
| 前端禁止 | HTML/JS文件中不得出现任何API Key |
| 环境变量 | 使用process.env读取敏感配置 |
| 日志脱敏 | 日志中不得打印完整API Key |

### B.7.3 会话安全

| 安全措施 | 说明 |
|---------|------|
| httpOnly Cookie | 禁止客户端JavaScript访问会话cookie |
| 会话过期 | 7天无活动自动过期 |
| 退出清理 | logout时彻底销毁session |

## B.8 前端页面

### B.8.1 login.html 登录页面

**文件路径**：`public/login.html`

**页面元素**：

| 元素 | 类型 | 说明 |
|------|------|------|
| 用户名输入框 | `<input type="text">` | name="username" |
| 密码输入框 | `<input type="password">` | name="password" |
| 登录按钮 | `<button>` | 提交表单 |
| 错误提示 | `<div>` | 显示登录失败信息 |

**页面逻辑**：

1. 监听登录按钮点击事件
2. 收集username和password
3. 发送POST请求到 `/login`
4. 登录成功则跳转到 `index.html`
5. 登录失败显示错误提示

**成功跳转**：

```javascript
if (response.success) {
  window.location.href = '/index.html';
}
```

### B.8.2 index.html 主页面

**文件路径**：`public/index.html`

**页面加载逻辑**：

```javascript
// 页面加载时检查登录状态
window.onload = async function() {
  try {
    const response = await fetch('/me');
    if (!response.ok) {
      // 未登录，跳转到登录页
      window.location.href = '/login.html';
    }
  } catch (error) {
    // 请求失败，跳转到登录页
    window.location.href = '/login.html';
  }
};
```

**退出登录按钮**：

- 提供退出按钮，调用 `/logout` 接口
- 退出成功后跳转到 `login.html`

### B.8.3 用户信息展示

**位置**：index.html页面头部或角落
**显示内容**：当前登录用户名
**退出按钮**：点击后退出登录

## B.12 设计原则

### B.12.1 核心理念

| 理念 | 说明 |
|------|------|
| 内部工具 | 不是面向用户的产品 |
| 快速可用 | 1天完成开发和部署 |
| 简单稳定 | 不追求花哨功能 |
| 够用就好 | 满足5-10人使用即可 |

### B.12.2 技术选型原则

- **能简单不复杂**：一个文件能解决的不拆成多个
- **能复用不重复**：利用现有框架特性
- **能不装就不装**：不需要的不引入

### B.12.3 后续扩展方向（可选）

如未来需要，可扩展：
- 用户管理界面
- 会话存储持久化
- 操作日志记录
- 多角色权限

**但当前阶段不要做，保持简单！**
